# Execution Plan: DI-011 — Audit Logging

## Delivery Item
[DI-011](../delivery-items/DI-011.md)

## Objective
Implement the complete audit logging system: AuditLog model, audit service, integration with all service methods and auth routes, admin-only UI with filtering and pagination.

## Approach

### 1. AuditLog Model (`src/app/models/audit_log.py`)
- Standalone model (not extending BaseModel — audit entries are permanent and never archived)
- Fields: `id`, `timestamp`, `user_id` (FK → users.id, nullable), `entity_type` (String), `entity_id` (Integer), `action` (String), `before_value` (JSON), `after_value` (JSON)
- Indexes on: `timestamp`, `(entity_type, entity_id)`, `user_id`
- `user_id` nullable to support system-generated events
- Use `db.JSON` for before/after values (maps to JSONB on PostgreSQL, text on SQLite)

### 2. `to_audit_dict()` on all auditable models
Each model gets a method that returns a dict of its auditable fields:
- `User`: email, full_name, role, is_archived, is_active
- `Team`: name, description, is_archived, is_active
- `Product`: name, description, is_archived, is_active
- `Repository`: name, url, description, team_id, template_id, is_archived, is_active
- `RepoTemplate`: name, description, is_archived, is_active

### 3. Audit Service (`src/app/services/audit_service.py`)
- `log(action, entity_type, entity_id, before=None, after=None, user_id=None)` function
- Called explicitly from service methods after successful db.session.commit()
- For login/logout: called after the action is confirmed

### 4. Service Integration
Add audit calls to:
- `user_service`: create_user (create), update_user (update + role_change if role changed), archive_user (archive), reactivate_user (update)
- `team_service`: create_team (create), update_team (update), archive_team (archive), reactivate_team (update)
- `product_service`: create_product (create), update_product (update), archive_product (archive), reactivate_product (update)
- `repository_service`: create_repository (create), update_repository (update), archive_repository (archive), reactivate_repository (update)
- `template_service`: create_template (create), update_template (update + template_change), archive_template (archive), reactivate_template (update), add_artifact (template_change), update_artifact (template_change), remove_artifact (template_change)
- `auth/routes.py`: login (login), logout (logout)

### 5. Audit Routes (`src/app/routes/audit.py`)
- `GET /audit/` — admin-only list page with filters
- Filter params: `entity_type`, `action`, `user_id`, `date_from`, `date_to`, `q` (text search)
- Pagination: 50 entries per page
- HTMX partial support

### 6. Templates
- `src/app/templates/audit/list.html` — full page with filter bar
- `src/app/templates/audit/partials/audit_table.html` — partial for HTMX updates

### 7. Migration
- New migration file adding the `audit_log` table

## Entity Type and Action Values
- entity_type: user, team, product, repository, template, session
- action: login, logout, create, update, archive, reactivate, role_change, template_change

## Audit Capture Strategy
- Before commit: capture `entity.to_audit_dict()` as `before`
- After commit: capture `entity.to_audit_dict()` as `after`
- For create: `before=None`, `after=entity.to_audit_dict()`
- For archive: capture before/after dict showing is_archived/is_active change
- For login/logout: `entity_type='session'`, `entity_id=user.id`, before/after=None
