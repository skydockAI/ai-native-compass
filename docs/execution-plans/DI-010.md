## Overview

This delivery item adds:
1. A functional dashboard with real entity counts (replacing placeholder `--` values)
2. A Users summary card on the dashboard
3. HTMX-driven filtering across all list pages
4. Reusable filter bar component (`components/filter_bar.html`)
5. Repository filtering by Product, Team, Template, and status
6. User list filtering by Role and status

No database schema changes are required — all work is in the service, route, and template layers.

## Implementation Steps

### 1. Dashboard Service (`src/app/services/dashboard_service.py`)

Create a new lightweight service that aggregates counts of active (non-archived) entities:
- `get_dashboard_counts()` → dict with keys: `products`, `teams`, `repositories`, `templates`, `users`
- Queries use `filter_by(is_archived=False).count()` on each model

### 2. Dashboard Route (`src/app/routes/dashboard.py`)

- Import and call `dashboard_service.get_dashboard_counts()`
- Pass the counts dict to the template

### 3. Dashboard Template (`src/app/templates/dashboard/index.html`)

- Replace `--` placeholder values with actual counts from the context variable
- Add a fifth summary card for Users (admin-only visibility matches sidebar)
- Add a "Users" card with the count

### 4. Repository Service (`src/app/services/repository_service.py`)

Extend `get_repositories()` to accept optional filter parameters:
- `product_id=None` — filter via JOIN on `product_repository` association table
- `team_id=None` — filter by `team_id` FK column
- `template_id=None` — filter by `template_id` FK column
- `include_archived` already exists; keep as-is

All filters are applied additively (AND logic).

### 5. Repositories Route (`src/app/routes/repositories.py`)

Update `index()`:
- Parse `product_id`, `team_id`, `template_id` from `request.args` (integer, default `None`)
- Pass filters to `repository_service.get_repositories()`
- Pass available filter options to template: all active products, teams, templates
- Detect HTMX request header (`HX-Request`) and render partial template for HTMX updates

### 6. Repository Templates

- Update `repositories/list.html` to include the filter bar and wrap the table in a `#repo-list-content` div
- Create `repositories/partials/repo_list.html` — just the table content for HTMX responses

### 7. User Service (`src/app/services/user_service.py`)

Extend `get_users()` to accept:
- `role=None` — if provided, filter by `UserRole` enum value

### 8. Users Route (`src/app/routes/users.py`)

Update `index()`:
- Parse `role` from `request.args`
- Pass to `user_service.get_users()`
- Detect HTMX request and render partial

### 9. User Templates

- Update `users/list.html` to include the filter bar with role dropdown and wrap the table in `#user-list-content`
- Create `users/partials/user_list.html` — just the table content for HTMX responses

### 10. Reusable Filter Bar Component (`src/app/templates/components/filter_bar.html`)

A generic Jinja2 component that accepts:
- `filter_action`: the URL to submit to (e.g. `url_for('repositories.index')`)
- `filter_fields`: list of field definitions (label, name, options, current_value)
- `hx_target`: CSS selector for HTMX swap target

### 11. Products, Teams, Templates List Pages

Enhance existing active/archived toggle to use HTMX partial updates:
- Wrap tables in `#<entity>-list-content` divs
- Create partials for each: `products/partials/product_list.html`, etc.
- Update route handlers to detect `HX-Request` and render partials

## HTMX Filtering Pattern

For each list page:
1. Filter bar renders as a `<form>` with `hx-get="<list_url>"`, `hx-target="#<entity>-list-content"`, `hx-push-url="true"`, `hx-trigger="change, submit"`
2. Form fields are `<select>` dropdowns; each filter change triggers an HTMX GET
3. Route handler checks `request.headers.get('HX-Request')`:
   - If truthy → render partial template (table only)
   - Otherwise → render full page template (includes layout, filter bar, table)
4. Full page renders include the partial via `{% include %}` so there's no HTML duplication
