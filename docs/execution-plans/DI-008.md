# Execution Plan: DI-008 — Template Change Propagation & Repository Duplication

## Delivery Item
[DI-008](../delivery-items/DI-008.md)

## Requirements Covered
- REQ-042 — Template Artifact Addition Propagation
- REQ-043 — Template Artifact Rename Propagation
- REQ-044 — Template Artifact Removal Behavior
- REQ-045 — List Option Editing and Deactivation
- REQ-052 — Repository Duplication

## Context

DI-007 delivered the core repository model (`Repository`, `RepositoryArtifactValue`) and CRUD routes. DI-005 delivered template and artifact management. DI-008 closes the gap between these two delivery items by:

1. Ensuring new artifacts propagate automatically to existing repositories using the template.
2. Changing `remove_artifact` from a hard-delete to a soft-delete so existing repository values are preserved.
3. Enabling repository duplication with all values pre-filled.

REQ-043 (rename propagation) and REQ-045 (list option editing/deactivation) are already fully handled by the existing data model (FK relationship for renames; `is_active` + in-use check already in DI-005 service). No new code is required for those two requirements.

## Detailed Plan

### 1. `template_service.py` — Soft-delete for `remove_artifact` (REQ-044)

Current behaviour: `db.session.delete(artifact)` — hard-deletes the row and cascades to `RepositoryArtifactValue`.

New behaviour:
```python
def remove_artifact(artifact_id):
    artifact = db.session.get(TemplateArtifact, artifact_id)
    if artifact is None:
        raise TemplateServiceError('Artifact not found.')
    artifact.is_active = False
    artifact.version += 1
    db.session.commit()
```

The `RepositoryArtifactValue` rows are preserved in the DB. The repository UI already filters on `is_active` via `selectattr('is_active')` in the Jinja2 templates and `[a for a in template.artifacts if a.is_active]` in the service.

### 2. `template_service.py` — Propagation in `add_artifact` (REQ-042)

After flushing the new artifact to get its `id`, insert an empty `RepositoryArtifactValue` for every active (non-archived) repository that uses this template, within the same transaction.

```python
def add_artifact(...):
    ...
    artifact = TemplateArtifact(...)
    db.session.add(artifact)
    db.session.flush()  # get artifact.id

    # Propagate: create empty artifact value rows for all active repos using this template
    from ..models.repository import Repository, RepositoryArtifactValue
    active_repos = Repository.query.filter_by(template_id=template_id, is_archived=False).all()
    for repo in active_repos:
        av = RepositoryArtifactValue(
            repository_id=repo.id,
            template_artifact_id=artifact.id,
        )
        db.session.add(av)

    db.session.commit()
    return artifact
```

All typed columns default to `NULL`, which is valid for all artifact types.

### 3. `repository_service.py` — `duplicate_repository` (REQ-052)

New function:
```python
def duplicate_repository(source_repo_id, name, url):
    """Duplicate a repository with a new name and URL.

    Copies:
    - All RepositoryArtifactValue rows (same template_artifact_id, same value columns)
    - All RepositorySharedAttributeValue rows (same attribute_id, same value)

    Does NOT copy:
    - product_repository links (REQ-052)
    """
```

The duplicate uses the source's `team_id` and `template_id`.

### 4. `routes/forms.py` — `RepositoryDuplicateForm`

Simple form with just `name`, `url`, and a hidden `source_repo_id`.

### 5. `routes/repositories.py` — Duplicate route

`GET /repositories/<id>/duplicate` — pre-fills the form with source repo data (name as "Copy of <name>", blank URL).
`POST /repositories/<id>/duplicate` — validates and calls `duplicate_repository`.

### 6. `templates/repositories/duplicate.html` — Duplicate form page

Simple form with name, url fields, pre-filled from source. Shows source repo name/template as read-only context.

### 7. Update `list.html` and `detail.html` — Duplicate button

Add a "Duplicate" button (Admin/Editor only) in the actions column / detail page header.
