# DI-008: Template Change Propagation & Repository Duplication

## Delivery Item ID
DI-008

## Title
Template Change Propagation & Repository Duplication

## Objective
Implement the automatic propagation of template artifact changes (additions, renames, removals) to all repositories using that template, along with list option editing/deactivation behavior and the repository duplication feature.

## Included Requirements
- **REQ-042** — Template Artifact Addition Propagation
- **REQ-043** — Template Artifact Rename Propagation
- **REQ-044** — Template Artifact Removal Behavior
- **REQ-045** — List Option Editing and Deactivation
- **REQ-052** — Repository Duplication

## Scope

### In Scope
- Template change propagation logic in `template_service.py`:
  - **Artifact addition**: When a new artifact is added to a template, automatically insert a `RepositoryArtifactValue` row (with null/empty value) for every active repository using that template
  - **Artifact rename**: Update `TemplateArtifact.name` — existing repository values remain attached via foreign key (no data migration needed)
  - **Artifact removal**: Set `TemplateArtifact.is_active = False` — existing values remain in DB but are excluded from active UI queries
- List option editing and deactivation in `template_service.py`:
  - Admin can edit list option labels (display value)
  - In-use options cannot be deleted but can be deactivated (`is_active = False`)
  - Deactivated options remain displayed on repositories that already selected them
  - Attempting to delete an in-use option prompts/suggests deactivation instead
  - Unused options can be permanently deleted
- Repository duplication in `repository_service.py`:
  - Copy all shared attribute values and template-derived artifact values from source repository
  - Duplicate uses the same template as the source
  - Duplicate requires a new unique URL
  - Product links are NOT copied to the duplicate
  - Admin and Editor access

### Out of Scope
- Template CRUD (already in DI-005)
- Repository CRUD (already in DI-007)
- Audit logging of propagation events (DI-011)

## Key Implementation Notes
- Artifact addition propagation should be performed within the same transaction as the artifact creation to ensure consistency
- Artifact removal uses soft-delete on `TemplateArtifact` (`is_active = False`), not hard delete — previously stored `RepositoryArtifactValue` rows are preserved for audit history
- Repository UI queries for artifact values should filter by `template_artifact.is_active = True` to hide removed artifacts
- Rename propagation is inherently handled by the relational model (artifacts are linked by ID, not name)
- List option deactivation: the UI should check `is_active` on list options when rendering dropdowns — deactivated options are hidden from new selection but still shown as the current value if already selected
- Repository duplication should pre-fill the create form with copied values, allowing the user to modify before saving

## Dependencies
- **DI-005** — Repo Template & Artifact Management (Template, TemplateArtifact, ArtifactListOption models)
- **DI-007** — Repository Core Management (Repository, RepositoryArtifactValue models, creation and editing logic)

## Definition of Done
- [ ] Adding an artifact to a template automatically creates empty artifact value rows for all active repositories using that template
- [ ] Renaming an artifact in a template updates the display name across all repositories without data loss
- [ ] Removing an artifact from a template hides it from repository UI but preserves stored values in the database
- [ ] List option labels can be edited by Admin
- [ ] In-use list options cannot be deleted; deactivation is offered instead
- [ ] Deactivated list options are hidden from new selections but shown on repositories that already use them
- [ ] Unused list options can be permanently deleted
- [ ] Repositories can be duplicated with all shared and artifact values copied
- [ ] Duplicated repositories require a new unique URL
- [ ] Product links are not copied during duplication
- [ ] Unit and integration tests cover propagation scenarios, list option lifecycle, and duplication
