# DI-007: Repository Core Management

## Delivery Item ID
DI-007

## Title
Repository Core Management

## Objective
Implement the Repository entity with the template-first creation workflow, editing rules, team assignment, URL uniqueness, immutable template assignment, shared attribute value entry, and template-derived artifact value management — delivering the central data entry experience of the application.

## Scope

### In Scope
- Repository model (`app/models/repository.py`):
  - Fields: `name`, `url` (unique), `description`, `team_id` (FK → Team), `template_id` (FK → RepoTemplate), `is_active`, `is_archived`, `created_at`, `updated_at`, `version`
  - Foreign key constraints to Team and RepoTemplate
- RepositoryArtifactValue model:
  - Fields: `repository_id` (FK), `template_artifact_id` (FK), `value_text`, `value_number`, `value_boolean`, `value_list_option_id` (FK), unique constraint on `(repository_id, template_artifact_id)`
- Repository service (`app/services/repository_service.py`):
  - Template-first creation workflow:
    1. User selects a template
    2. System loads shared fields + template artifacts dynamically
    3. User fills in all fields and submits
  - URL uniqueness validation (including archived repositories)
  - Template assignment at creation (immutable after creation)
  - Team assignment (mandatory, changeable after creation, only active teams)
  - Artifact value storage per the typed column strategy (value_text, value_number, value_boolean, value_list_option_id)
  - Shared attribute value management (custom EAV attributes)
  - Repository editing: update shared values, artifact values; template remains read-only
- Repository routes (`app/routes/repositories.py`):
  - Repository list (active/archived filter), detail, create, edit, archive
  - HTMX endpoint for dynamic template selection (loads artifact fields)
  - Admin and Editor access for create/edit/archive; Viewer read-only
- Repository Jinja2 templates:
  - Repository list page with active/archived toggle
  - Repository detail page showing shared attributes and artifact values
  - Repository create form with template selector (HTMX-driven dynamic form)
  - Repository edit form (template read-only, shared fields and artifacts editable)
  - HTMX partials: `artifact_fields.html` (loaded after template selection)
- product_repository association table (M:N — table created here, linking UI in DI-009)
- Alembic migrations for Repository, RepositoryArtifactValue, and product_repository tables

### Out of Scope
- Product-Repository linking UI (DI-009)
- Template change propagation to existing repositories (DI-008)
- Repository duplication (DI-008)
- Repository archive blocking by product links (DI-009)
- Dashboard and filtering views (DI-010)
- Audit logging (DI-011)

## Key Implementation Notes
- The creation workflow is template-first: the user selects a template, then HTMX dynamically loads the artifact fields for that template alongside shared attribute fields
- `template_id` is set once at creation and becomes read-only — the edit form must not allow changing it
- Every repository must belong to exactly one team; only active teams appear in the team dropdown
- Repository URL uniqueness is enforced at the database level (includes archived repositories)
- Artifact values use the typed column approach per the architecture (Section 4.2):
  - Document/Skill/Agent → `value_text` (Yes/No/N/A)
  - Other:text → `value_text`
  - Other:number → `value_number`
  - Other:boolean → `value_boolean` (nullable tri-state)
  - Other:list → `value_list_option_id` (FK to ArtifactListOption)
- On creation, a RepositoryArtifactValue row is inserted for each active artifact in the selected template (with null/empty default values)
- Shared data is stored once at the repository level, not per product link (REQ-030) — this is inherent in the data model
- Required artifact validation (REQ-040) should be enforced on save
