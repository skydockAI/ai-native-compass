# DI-002: Authentication & User Session Management

## Delivery Item ID
DI-002

## Title
Authentication & User Session Management

## Objective
Implement the complete authentication system including local email/password login, secure session management, admin user seeding from environment, and seeded admin protection — with the authentication module designed for future SSO extensibility.

## Included Requirements
- **REQ-001** — Local Email/Password Authentication
- **REQ-002** — Active User Login Restriction
- **REQ-003** — Secure Password Storage
- **REQ-004** — Session-Based Authentication
- **REQ-007** — Immediate Access Denial for Archived/Removed Users
- **REQ-008** — Admin User Seeding from Environment
- **REQ-009** — Seeded Admin Protection
- **REQ-010** — SSO-Ready Authentication Design
- **REQ-068** — Security Best Practices (authentication-related aspects)

## Scope

### In Scope
- User model with fields: `email`, `full_name`, `password_hash`, `role` (enum: admin/editor/viewer), `is_active`, `is_archived`, `is_seeded`, `created_at`, `updated_at`, `version`
- Authentication module (`app/auth/`) with pluggable design:
  - `local.py` — email/password verification, password hashing (Werkzeug/bcrypt)
  - `routes.py` — `/login` and `/logout` endpoints
  - `__init__.py` — exports `authenticate_user()` for swappability
- Flask-Login integration for session management
- Login page UI (email + password form, generic error messages)
- Password hashing with minimum 8-character enforcement
- Active user check on login (reject archived/inactive users)
- Session validation on every request via `before_request` hook (REQ-007)
- Admin seeding utility (`app/utils/seed.py`): reads `ADMIN_SEEDS` from `.env`, creates admin users with `is_seeded=True`
- Seeded admin protection: prevent demotion, archive, or deletion of seeded admins (enforced in service layer)
- Secure cookie configuration (HTTP-only, secure flag)
- CSRF protection on login form
- Alembic migration for the User table

### Out of Scope
- User management CRUD UI (DI-003)
- Role-based access control decorators (DI-003)
- Password change and reset features (DI-003)

## Key Implementation Notes
- The auth module must be isolated from authorization logic to support future SSO replacement (REQ-010)
- `authenticate_user(email, password)` should be the single entry point — a future SSO module replaces this function
- Login errors must not reveal whether the email or password was incorrect (generic message only)
- The `before_request` hook must check `is_active` and `is_archived` on every request to enforce immediate access denial
- Admin seeding runs during `create_app()` startup, after migrations
- Seeded admin protection is enforced at the service layer, not just the UI
- Email uniqueness is case-insensitive and enforced at the database level

## Dependencies
- **DI-001** — Project Foundation & Infrastructure Setup (Flask app, database, migrations, base UI)

## Definition of Done
- [ ] Users can log in with email and password, receiving a session cookie
- [ ] Login page displays generic error messages on authentication failure
- [ ] Archived or inactive users are rejected at login
- [ ] Active sessions are invalidated immediately when a user is archived/deactivated
- [ ] Passwords are securely hashed; minimum 8-character length is enforced
- [ ] Admin users are seeded from `ADMIN_SEEDS` environment variable on startup
- [ ] Seeded admin users are flagged with `is_seeded=True` and cannot be demoted or deleted
- [ ] Logout clears the session and redirects to the login page
- [ ] Auth module is cleanly separated from business logic (SSO-ready)
- [ ] User table migration is applied with all required columns and constraints
- [ ] Unit and integration tests cover login, logout, session validation, and admin seeding
