# DI-011: Audit Logging

## Delivery Item ID
DI-011

## Title
Audit Logging

## Objective
Implement the complete audit logging system — capturing all required events (login, logout, create, update, archive, role change, template change), storing structured before/after values, enforcing admin-only access, and providing a filterable, searchable audit log UI.

## Included Requirements
- **REQ-048** — Permanent Audit History Preservation
- **REQ-059** — Audit Log Events
- **REQ-060** — Audit Log Entry Structure
- **REQ-061** — Audit Log Access Control
- **REQ-062** — Audit Log UI

## Scope

### In Scope
- AuditLog model (`app/models/audit_log.py`):
  - Fields: `id`, `timestamp`, `user_id` (FK, nullable), `entity_type` (string enum), `entity_id`, `action` (string enum), `before_value` (JSONB), `after_value` (JSONB)
  - Indexes on `timestamp`, `(entity_type, entity_id)`, `user_id`
- Audit service (`app/services/audit_service.py`):
  - `log()` function for recording audit events
  - Called explicitly from service methods (not via ORM hooks)
  - Before/after value capture as field-level diffs (not full entity snapshots)
- `to_audit_dict()` method on all auditable models (User, Team, Product, Repository, RepoTemplate)
- Integrate audit logging into all existing service methods:
  - Login/logout events (in auth routes)
  - Create, update, archive events for: Users, Teams, Products, Repositories, Templates
  - Role change events (user role modification)
  - Template change events (artifact add/rename/remove affecting live repos)
- Audit log routes (`app/routes/audit.py`):
  - Audit log list page (Admin-only)
  - Filtering by entity type, action type, user, and date range
  - Text search across entity identifiers
  - Pagination
- Audit log Jinja2 templates:
  - Audit log list page with filter controls
  - Reverse chronological ordering (newest first)
  - Expandable detail view for before/after values
- Audit log access control:
  - Admin-only route access (server-side enforcement)
  - Sidebar "Audit" link hidden for non-Admin users
- Permanent preservation: audit log entries are never deleted
- Alembic migration for AuditLog table

### Out of Scope
- Audit log export (deferred)
- Automatic purge or retention policies (not in v1 per REQ-048)
- Audit log editing, deletion, or restore capabilities

## Key Implementation Notes
- Audit logging is a service-level concern using explicit calls (architecture Section 8.1), not ORM event hooks
- Before/after values capture field-level changes, not full entity snapshots
- `user_id` is nullable to support system-generated events (e.g., admin seeding on startup)
- `entity_type` values: user, team, product, repository, template, session
- `action` values: login, logout, create, update, archive, role_change, template_change
- JSONB columns for before/after values enable flexible queries and PostgreSQL native JSON operations
- Audit entries must never be deleted — no hard delete, no archive, no purge
- This DI requires touching all existing service files to add audit logging calls
- The audit log UI is view-only in v1 (no edit/delete/restore capabilities)
- Text search should cover entity identifiers and JSONB value content
