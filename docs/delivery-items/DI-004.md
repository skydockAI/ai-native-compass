# DI-004: Team Management

## Delivery Item ID
DI-004

## Title
Team Management

## Objective
Implement the Team entity with full CRUD operations, name uniqueness enforcement, and archive protection that prevents archiving a team with active repositories assigned to it.

## Included Requirements
- **REQ-019** — Team Entity Definition
- **REQ-020** — Team Name Uniqueness
- **REQ-022** — Team Archive Protection
- **REQ-049** — Dependency-Based Archive Blocking (Team portion)

## Scope

### In Scope
- Team model (`app/models/team.py`):
  - Fields: `name`, `description`, `is_active`, `is_archived`, `created_at`, `updated_at`, `version`
  - Database-level unique constraint on `name`
- Team service (`app/services/team_service.py`):
  - Create, read, update, archive operations
  - Name uniqueness validation with user-friendly error messages
  - Archive protection: check for active repositories before allowing archive
  - Return list of blocking repositories when archive is denied
- Team routes (`app/routes/teams.py`):
  - Team list (active/archived filter), create, edit, archive
  - Admin and Editor access for create/edit/archive; Viewer read-only
- Team Jinja2 templates:
  - Team list page with active/archived toggle
  - Team create/edit form
  - Archive confirmation showing dependent repositories (if any)
- Alembic migration for the Team table

### Out of Scope
- Repository-Team assignment UI (DI-007 — implemented when Repository entity is built)
- Audit logging of team actions (DI-011)

## Key Implementation Notes
- Team name uniqueness is enforced at the database level (unique constraint) and validated in the service layer
- Archive blocking must query active (non-archived) repositories assigned to the team
- The UI should display which active repositories block the archive action, with a clear message
- Admins and Editors can manage teams; Viewers have read-only access
- Follow the same list view pattern established in DI-003 (active/archived filter toggle)

## Dependencies
- **DI-001** — Project Foundation & Infrastructure Setup (base model mixin, migrations)
- **DI-003** — User Management & RBAC (role decorators, archived entity visibility pattern)

## Definition of Done
- [ ] Teams can be created, viewed, edited, and archived by Admin and Editor users
- [ ] Viewer users can view teams but cannot create, edit, or archive
- [ ] Team name uniqueness is enforced at the database level
- [ ] Duplicate team name attempts show a clear error message
- [ ] Archiving a team with active repositories is blocked with a descriptive message listing the blocking repositories
- [ ] Teams with no active repositories can be archived successfully
- [ ] Team list page supports active/archived filter toggle
- [ ] Alembic migration for the Team table is present and applies cleanly
- [ ] Unit and integration tests cover CRUD operations, uniqueness, and archive protection
