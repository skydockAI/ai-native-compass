# DI-003: User Management & Role-Based Access Control

## Delivery Item ID
DI-003

## Title
User Management & Role-Based Access Control

## Objective
Implement the full user management UI (admin-only CRUD), the three-role RBAC system with route-level enforcement decorators, self-service password change, admin password reset, and the archived entity visibility pattern that will be reused across all subsequent entity list views.

## Scope

### In Scope
- Authorization module (`app/authz/`):
  - `@login_required` integration with Flask-Login
  - `@role_required('admin')`, `@role_required('admin', 'editor')` decorators
  - Route-level and template-level role enforcement
- User management routes (`app/routes/users.py`):
  - User list (admin-only), create, edit, archive, reactivate
  - Role assignment and change
- User service (`app/services/user_service.py`):
  - CRUD operations with validation
  - Email uniqueness enforcement (case-insensitive, including archived users)
  - Seeded admin protection checks
  - Role change logic
- Self-service password change (current password + new password with validation)
- Admin password reset for any user account
- User management Jinja2 templates:
  - User list page with active/archived filter toggle
  - User create/edit form
  - Password change form
- Archived entity UI visibility pattern:
  - Default list views show only active entities
  - Optional filter/toggle to show archived entities separately
  - Archived entities hidden from dropdowns and selection controls
- Role-aware sidebar: Users and Audit links visible only to Admin role
- Conditional UI element rendering based on `current_user.role`

### Out of Scope
- Audit logging of user actions (DI-011)
- Dashboard views (DI-010)

## Key Implementation Notes
- Role is stored as a database enum (`admin`, `editor`, `viewer`) on the User model — enforced at DB level
- Each user has exactly one role; changing a role replaces the previous one
- Authorization decorators must be applied to every route handler; UI hiding is a convenience only
- Email uniqueness check must include archived users (DB-level unique constraint)
- The archived entity visibility pattern (show/hide toggle, filtering) should be built as a reusable approach for all entity list pages
- Password change requires current password verification; admin reset does not
- Seeded admins cannot be demoted, archived, or deleted — enforced in service layer with clear error messages
